1. Executive Verdict

Do Not Advance

The repository is a serious Week 1 research harness, but it is not phase-ready yet. The strongest positives are that it does persist a useful artifact bundle, enforces an OOS embargo before the LangGraph run, implements validation-only threshold tuning in the main path, and has at least some explicit tests for signal-delay parity and purge gaps.  ￼

The blockers are material. First, the phase-gate / readiness logic is wrong: the development plan says future learning work should start only after stable snapshot-based runs, reproducible held-out and walk-forward metrics, persisted artifacts across multiple windows, and clear regression tracking, yet the current run’s readiness artifact still says "phase_gate_decision": "advance" and "fine_tuning_gate_open": true despite triple-negative validation, walk-forward, and held-out Sharpe.  ￼ Second, there is at least a split-boundary contamination issue: target_up is defined with close.shift(-1), so the final training row uses the first validation close as its label, which violates the plan’s claim that the train split is used only for fitting and validation is used only for calibration.  ￼ Third, the repository has a source/artifact timing ambiguity around execution timestamps: the persisted trades in this run show bar_timestamp == execution_timestamp, while currently visible source snippets suggest some runtime stamping logic may set execution timestamps one bar ahead when a delayed signal is applied; that must be reconciled at the exact run commit before trusting regression comparisons.  ￼

Performance itself being bad is not a blocker by itself for a research baseline. But bad performance plus a false-open readiness gate plus unresolved temporal edge cases is a blocker. Also, the user-provided run context does not match the actual artifact bundle: the artifacted run is 5,000 bars on a snapshot from April 6 to April 9, 2026, not 1,000 bars from April 1 to April 11.  ￼

2. What Stage the Repository Is Actually In

This is a research baseline, not yet a reproducible research baseline.

Why:
	•	It does implement the intended prototype stack: feature engineering, LightGBM directional baseline, validation threshold calibration, walk-forward benchmarking, held-out LangGraph simulation, and artifact persistence.  ￼
	•	It has enough tests to show the authors are thinking about timing semantics and persistence.  ￼
	•	But it is not yet reproducible at the phase-gate standard because the gate requirements explicitly call for reproducible metrics across multiple historical windows and benchmark regression tracking, and the current evidence is just one snapshot-backed run plus one walk-forward benchmark inside that run.  ￼
	•	It is nowhere near a paper-trading or production candidate; the README itself says it is not claiming live trading or paper trading.  ￼

4. End-to-End Timeframe-Sensitive Logic Audit

Data load

Intended behavior: load exchange or snapshot data, sort by timestamp, build features, then compute metadata/splits.

Actual behavior: load_historical_data loads either a snapshot or exchange pull, sorts by timestamp, passes through build_features, and records dataset metadata. compute_split_metadata then creates train/validation/OOS boundaries, with required_embargo_gap = max(purge_bars, max_feature_lag + 1).  ￼

Temporal validity: mostly valid.

Discrepancy / hidden assumption: the current run is snapshot-based and therefore benchmark-eligible, but the prompt’s claimed dataset span and bar count do not match the actual artifacts. The artifacted run is 5,000 bars, train 3,000, validation 1,000, OOS 200.  ￼

Feature build

Intended behavior: all features should be backward-looking at each base bar.

Actual behavior: base features are computed on the raw 15m stream; higher-timeframe features are built with group_by_dynamic(..., closed="left", label="left"), then all HTF feature columns are shifted by one bucket before join_asof(..., strategy="backward"). That is conservative and intentionally restricts each base bar to completed higher-timeframe buckets strictly before the base timestamp.  ￼

Temporal validity: good.

Discrepancy / hidden assumption: this is correct for 15m because the default MTF ladder is stepped up to 1h/4h/1d; the development plan still says “BTC 1m data,” so the planning doc is stale and could mislead future contributors.  ￼

Target label

Intended behavior: predict next-bar direction causally.

Actual behavior: target_up = (close.shift(-1) > close).cast(Int8).fill_null(0).  ￼

Temporal validity: valid as a supervised label definition.

Discrepancy / hidden assumption: because labels are precomputed on the full frame before slicing, the last training row uses the first validation close as its label, and similarly the last validation row uses the next bar beyond the validation slice. That is only a one-bar horizon issue, but it violates a strict reading of “train split used only for fitting” and “validation split used only for calibration.” This is not feature leakage, but it is split-boundary label contamination.  ￼

Split construction

Intended behavior: train, validation, and OOS should be temporally separated; OOS should be protected by purge/embargo.

Actual behavior: oos_start = max(validation_end + required_embargo_gap, warmup_bars) with required_embargo_gap = max(purge_bars, max_feature_lag + 1). In the current run artifact that becomes required_embargo_gap = 17, validation_end = 4000, oos_start = 4017.  ￼

Temporal validity: good for validation→OOS separation.

Discrepancy / hidden assumption: there is no analogous embargo between train and validation despite the one-bar target horizon. That is the main split-quality weakness.  ￼

Model training

Intended behavior: fit only on train.

Actual behavior: train_lightgbm_baseline slices [train_start, train_end) and fits only that slice.  ￼

Temporal validity: mostly valid.

Discrepancy / hidden assumption: still inherits the boundary-label issue above.

Validation calibration

Intended behavior: thresholds selected on validation only; OOS untouched.

Actual behavior: calibrate_thresholds takes validation_df and validation probabilities only, then maximizes validation Sharpe with simulate_policy. The metadata explicitly says "affects_future_oos_only": true.  ￼

Temporal validity: good.

Discrepancy / hidden assumption: the best validation score in the current run is strongly negative, so the calibration process is working mechanically but is not finding a usable policy. That is a research-result problem, not by itself an engineering defect.  ￼

Walk-forward folds

Intended behavior: purge gaps between validation and test; all folds strictly before held-out OOS region.

Actual behavior: fold builder uses test_start_actual = validation_end + purge_bars, and tests assert test_start - validation_end == purge_bars. Folds are built while test_start + test_bars < oos_start, so the benchmark is pre-OOS.  ￼

Temporal validity: good.

Discrepancy / hidden assumption: again, no explicit train→validation embargo despite one-bar labels. Also, the visible artifact evidence shows many folds and very negative fold Sharpe values, but the readiness logic still marks walk-forward evidence as sufficient without showing a richer sufficiency contract in the benchmark artifact itself.  ￼

OOS graph execution

Intended behavior: forward-only cursor through held-out bars, with signal delay preserved in runtime.

Actual behavior: graph is data -> runtime -> predict -> risk -> decision -> evaluate -> optimize; data_node enforces the minimum OOS cursor at validation_end + max(purge, required_embargo_gap). Tests assert runtime and batch simulation parity for returns/equity/trade timestamps.  ￼

Temporal validity: broadly good.

Discrepancy / hidden assumption: there is a source/artifact ambiguity around execution timestamp stamping. The run artifact’s trades have bar_timestamp == execution_timestamp, and the report says execution occurs at close(t + delay).  ￼ But currently visible source snippets suggest some runtime code may stamp execution timestamps one bar ahead when the pending signal is applied.  ￼ That ambiguity must be resolved against the exact run commit before treating trade logs as regression-grade truth.

Metrics / artifacts

Intended behavior: persist metrics and diagnostics consistently enough for regression tracking.

Actual behavior: the run bundle includes report.md, report.json, run_metrics.json, benchmark_metrics.json, split_metadata.json, strategy_params.json, returns, equity_curve, trades, completed_trades, event counts, and logs. Tests explicitly assert key files are written and that trade-count mismatches should raise.  ￼

Temporal validity: good.

Discrepancy / hidden assumption: artifact completeness for a single run is decent; artifact completeness for the plan-defined phase gate is not. There is no multi-window regression ledger, and the readiness decision is therefore too optimistic.  ￼

6. Evidence From Current Run Artifacts

The current run is mechanically coherent but economically poor. Validation Sharpe is about -18.34, walk-forward mean Sharpe is about -57.34, and held-out Sharpe is about -9.32. That pattern means the model/threshold stack is not just underperforming on held-out OOS; it is also underperforming in validation and across the pre-OOS walk-forward benchmark. This looks much more like a weak baseline on this window than a classic hidden-overfit case where validation is strong and OOS collapses.  ￼

The held-out artifact bundle is internally more consistent than the prompt summary suggests. run_metrics.json reports transition_count = 9 and completed_trade_count = 4; completed_trades.json contains 4 rows; event counts show trade_opened = 5 and trade_closed = 4, which is consistent with one open trade remaining at the end of the OOS window. The report headline also shows Transition Count 9 and Completed Trade Count 4.  ￼

So the current artifact evidence does not support the specific “trade_count mismatch 43 vs 87” concern stated in the prompt. For this run, the main KPI counts appear internally aligned. The bigger artifact problem is not count inconsistency; it is that the readiness artifact still declares "advance" and opens the fine-tuning gate despite explicitly bad validation, benchmark, and held-out outcomes.  ￼

The overfit warning logic is also too shallow. The report warns only that both held-out and walk-forward Sharpe are negative, which is true, but it does not escalate that the entire evidence stack is uniformly negative and therefore fails the plan’s “benchmark stability before next phase” intent.  ￼
