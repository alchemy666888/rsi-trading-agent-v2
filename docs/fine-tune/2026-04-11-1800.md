# Repository Alignment Remediation Plan

## Summary
Bring the repository into honest alignment with its actual stage by converting it from a loosely marketed “self-improving RSI agent” into a validated Week 1 research prototype with clean out-of-sample evaluation, explicit scope, and real safety boundaries.

The plan assumes the intended near-term target is the documented Week 1 milestone: feature engineering + LightGBM baseline + walk-forward validation, not RL, LoRA, RAG, live trading, or persistent agent memory. Any code or docs implying later-stage capabilities must either be implemented for real or downgraded to explicit placeholders.

## Implementation Changes
### 1. Re-baseline the project around a truthful current stage
- Rewrite the public project description to state that the current system is a LangGraph-orchestrated BTC simulation prototype with LightGBM-based directional prediction and threshold tuning.
- Remove or explicitly label as future work any claims of LoRA fine-tuning, RL policy learning, RAG, persistent memory, human-in-the-loop review, paper trading, or live execution.
- Decide the strategy label and make it consistent everywhere:
  - Default: keep the repo as a model-based BTC trading prototype and stop calling it an RSI agent.
  - If the RSI branding must remain, then narrow the runtime strategy so RSI is the primary signal and LightGBM is optional experimental mode.

### 2. Fix the evaluation architecture before any further tuning
- Split the historical dataset into explicit `train`, `validation`, and `out_of_sample_run` segments.
- Train the baseline model only on the training segment.
- Use the validation segment only for threshold calibration and hyperparameter search.
- Run the LangGraph cycle loop only on the out-of-sample segment; it must never consume bars seen during training or threshold search.
- Add a one-bar action delay to all trading decisions so a prediction made from bar `t` can only affect execution from `t+1`.
- Replace the current walk-forward implementation with purged time-series folds:
  - expanding and rolling modes remain
  - each fold has a purge gap between train and test
  - each fold calibrates its own thresholds from its own train/validation data
  - fold metrics are aggregated separately from the main run metrics
- Treat walk-forward as the primary benchmark and the main cycle loop as a single held-out simulation, not vice versa.

### 3. Refactor runtime responsibilities into clear modules
- Break the monolithic node implementation into behavior-level modules:
  - data loading and snapshot management
  - feature engineering
  - model training and inference
  - decision policy
  - risk controls
  - evaluation and reporting
  - artifact persistence
- Keep the LangGraph topology small and explicit:
  - `data -> predict -> risk -> decide -> evaluate -> optimize`
  - `optimize` remains cyclic only for allowed online parameter updates, not model retraining unless explicitly supported
- Define state fields with tighter meaning:
  - `historical_data` becomes immutable run snapshot
  - `cursor` indexes only the out-of-sample loop
  - `prediction` contains timestamped signal metadata
  - `strategy_params` contains only currently active policy/risk parameters
  - `performance` is split into `run_metrics` and `benchmark_metrics`
- Remove dead or misleading state fields unless implemented for real.

### 4. Add a real risk and execution-simulation layer
- Insert a dedicated risk node that enforces:
  - max position size
  - max turnover per bar
  - stop-loss / take-profit policy if enabled
  - max drawdown pause
  - optional no-trade zones during high volatility
- Replace implicit full-notional exposure with explicit position sizing rules:
  - default: unit long/short position sizing with configurable cap
  - transaction costs remain configurable and applied on position changes
- Add a paused state so the loop can halt when drawdown or risk limits are breached.
- Keep execution simulated only; no broker order placement code should exist in the current stage.

### 5. Make optimization honest and stage-appropriate
- Scope “optimization” to what actually exists now:
  - threshold tuning
  - LightGBM hyperparameter tuning
  - feature-set ablations / toggles
  - risk-parameter tuning
- Remove the fake LoRA update hook or replace it with a no-op that is clearly documented as future work and excluded from reports.
- Keep replay buffer only if it becomes a persisted research artifact for later RL; otherwise rename it to `trade_history_buffer` to avoid implying learning capability.
- Ensure optimization events record:
  - data split used
  - objective definition
  - chosen parameters
  - whether the event affected only future out-of-sample bars

### 6. Add persistence, auditability, and reproducibility
- Add offline dataset snapshot support so runs can execute without live exchange dependency.
- Each run writes a versioned artifact bundle containing:
  - dataset metadata and date range
  - config used
  - model params
  - threshold params
  - trade log
  - equity curve
  - walk-forward fold metrics
  - SHAP outputs
  - final report
- Generated reports must distinguish clearly between:
  - held-out run metrics
  - walk-forward benchmark metrics
  - optimization history
- Add `.gitignore` and remove IDE/bytecode artifacts from tracked state in future cleanup.

### 7. Define fine-tuning readiness as a future gated milestone
- Do not implement model fine-tuning in the current stage.
- Add a documented gate for future learning work:
  - stable offline dataset snapshots
  - clean benchmark harness
  - reproducible OOS metrics over multiple windows
  - persisted trade and feature artifacts
- Treat future “fine-tuning” tracks as separate milestones:
  - parameter tuning
  - policy tuning
  - model replacement
  - live paper-trading readiness

## Public Interfaces / Types / Outputs
- `config.yaml` should gain explicit split and benchmark settings:
  - dataset source mode: `exchange` or `snapshot`
  - train/validation/test boundaries or ratios
  - purge gap
  - signal delay
  - risk limits
  - artifact output path
- Reports should expose separate sections for:
  - held-out simulation results
  - walk-forward benchmark results
  - optimization provenance
  - active risk rules
- State/type changes:
  - remove ambiguous learning-oriented fields unless used
  - introduce explicit benchmark and artifact metadata fields
  - add paused/risk status fields
- Runtime contract:
  - one command runs a reproducible offline experiment
  - optional live fetch only builds a snapshot, not the canonical benchmark

## Test Plan
- Data split correctness:
  - model never trains on bars used in held-out run
  - threshold tuning never uses held-out bars
  - purge gap is enforced in every fold
- Signal timing:
  - predictions from bar `t` cannot execute on bar `t`
  - transaction costs apply only on position changes
- Risk controls:
  - drawdown breach pauses the run
  - position cap prevents oversizing
  - no-trade regime correctly suppresses entries when configured
- Benchmark integrity:
  - expanding and rolling folds are reproducible
  - fold-local thresholds differ when data differs
  - reported aggregate metrics match fold outputs
- Artifact and reporting:
  - every run writes a complete artifact bundle
  - report sections label OOS vs walk-forward correctly
  - optimization records include provenance
- Regression coverage:
  - offline snapshot run completes without network
  - baseline report generation succeeds on a fixed small fixture dataset

## Sprint Breakdown
### Phase 1: Immediate alignment fixes
- Rewrite docs and naming to match actual scope.
- Introduce offline snapshot input and explicit split config.
- Move the main cycle loop fully onto held-out data.
- Add one-bar signal delay and correct cost application.
- Rebuild walk-forward with purge gap and fold-local calibration.

### Phase 2: Architecture cleanup
- Modularize the current node logic by subsystem.
- Add the risk node and paused state.
- Remove or rename fake learning constructs.
- Separate run metrics from benchmark metrics.

### Phase 3: Evaluation and tuning readiness
- Persist artifact bundles for every run.
- Add tests for leakage, timing, and risk behavior.
- Expand optimization to include only legitimate tunables.
- Produce a benchmark-driven report suitable for comparing revisions.

### Phase 4: Optional performance improvements
- Cache engineered features per snapshot.
- Add feature ablation switches.
- Add multiple historical windows and stress-period benchmark presets.
- Prepare a future milestone doc for paper trading and real learning work.

## Acceptance Criteria
- The repo no longer claims capabilities not present in code.
- The main reported performance comes from held-out data only.
- Walk-forward evaluation is purged, delayed, and reproducible.
- Risk controls can pause the simulation and are visible in outputs.
- Each run produces a complete artifact bundle with provenance.
- The codebase has automated tests covering leakage, timing, and risk.
- Fine-tuning is documented as gated future work, not implied current behavior.

## Assumptions
- Near-term target stage is the documented Week 1 milestone, not Weeks 2-6.
- Live trading, paper trading, RL, LoRA, RAG, vector DB memory, and human review stay out of scope for this remediation pass.
- The recommended default is to rebrand the current system as model-based rather than preserve the RSI-first label.
- The implementation should prioritize research integrity and truthful reporting over adding new trading features.
