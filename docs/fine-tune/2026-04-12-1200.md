1. Executive Verdict

Do Not Advance

This repository is not ready to move to the next phase. The biggest blocker is a real temporal-integrity defect: the model is trained on target_up labels that are precomputed on the full dataset, so the last training row in each split uses the next close from the following split. That contaminates both the main train/validation boundary and the walk-forward folds. src/agents/features.py constructs target_up with close.shift(-1), and src/agents/modeling.py plus src/agents/evaluation.py slice train windows without dropping the boundary-crossing label row.  ￼  ￼  ￼

There is also a simulation-parity defect: max_drawdown_pause is enforced in the LangGraph runtime via evaluate_risk, but it is not implemented in simulate_policy, which is the engine used for validation threshold calibration and walk-forward benchmarking. That means the calibration path, benchmark path, and held-out runtime are not actually evaluating the same policy.  ￼  ￼

On top of that, the current run evidence is internally suspect. The trade-count mismatch you called out, the absence of an overfit/negative-performance warning despite both held-out and walk-forward Sharpe being negative, and the split-count numbers you supplied do not line up with what the current artifact writer and split logic would permit. persist_run_artifacts now explicitly rejects legacy trade_count/win_rate schemas and asserts headline/report consistency, while compare_benchmark_metrics would emit a warning when both Sharpe values are negative. compute_split_metadata with the current config also does not yield the OOS bar count you reported.  ￼  ￼  ￼

So the problem is not merely poor strategy performance. Poor metrics would be acceptable for a Week 1 research baseline. The blocking issue is that the repository does not yet provide trustworthy, temporally clean, artifact-consistent evidence for a phase-gate decision. The codebase should remain in the current phase until the leakage, simulation-parity, and artifact-integrity issues are resolved.  ￼

2. What Stage the Repository Is Actually In

This is best described as an early research baseline with partial reproducibility scaffolding, not a reproducible research baseline.

The repo has the right broad components for a Week 1 prototype: snapshot/exchange loading, broad feature engineering, a LightGBM directional model, validation-time threshold calibration, a LangGraph OOS loop, walk-forward benchmarking, and artifact persistence.  ￼  ￼

But it falls short of a true reproducible research baseline for three reasons. First, the train-boundary label leakage means the core benchmarking evidence is not temporally clean. Second, runtime-vs-benchmark risk semantics are not fully aligned because max_drawdown_pause exists only in the runtime path. Third, the run artifact evidence you provided appears inconsistent with the current source implementation. That combination is below the bar for opening any next-phase work, and far below paper-trading or production candidacy.  ￼  ￼  ￼  ￼

4. End-to-End Timeframe-Sensitive Logic Audit

Data load

Intended behavior: Load a frozen snapshot or exchange pull, sort chronologically, then build features on 15-minute BTC data.
Actual behavior: load_historical_data reads a snapshot or fetches exchange OHLCV, sorts by timestamp, hashes the raw market frame, then passes the data into build_features. Snapshot mode is recorded in dataset metadata and marked benchmark-eligible.  ￼

Temporal validity: Mostly valid.
Hidden assumption: Exchange mode and snapshot mode share the same downstream path, but only snapshot mode is considered regression-eligible. That matches the plan.  ￼  ￼

Feature build

Intended behavior: Build strictly backward-looking base-timeframe and higher-timeframe features.
Actual behavior: build_features computes returns, TA-Lib indicators, lags, rolling stats, ETH-relative features, volatility regime flags, and target_up. Multi-timeframe features are built with group_by_dynamic, then all MTF feature columns are shifted by 1 bucket before the backward as-of join. That design is meant to ensure a 15-minute bar at time t only sees the most recent completed 1h/4h/1d bucket.  ￼

Temporal validity: Base indicators and lag/rolling expressions are backward-looking. The MTF implementation is also causally valid for the current 15-minute use case because the pre-join shift(1) prevents the current incomplete higher-timeframe bucket from leaking into the base bar.  ￼

Discrepancy / hidden assumption: The implementation is hard-wired to a one-bar label horizon and to a small set of timeframes. It is correct enough for 15m, but fragile if contributors change the base timeframe or add non-divisor intervals without re-auditing the resampling/join logic. Also, all missing float/int values are zero-filled globally, which is acceptable for a Week 1 prototype but can mask warmup incompleteness.  ￼

Target label

Intended behavior: Directional one-step label: next close greater than current close.
Actual behavior: target_up = (close.shift(-1) > close).cast(Int8).fill_null(0).  ￼

Temporal validity: The label definition itself is standard and causally fine if split boundaries are handled correctly.
Discrepancy / hidden assumption: They are not. Because target_up is computed on the entire feature frame before splitting, the last row of any train slice uses the next close from the next segment. That leaks one-bar future information from validation into training and from later fold windows into earlier fold training. This is the repository’s most important temporal defect.  ￼  ￼  ￼

Split construction

Intended behavior: Train for fitting, validation for threshold calibration, OOS held out for LangGraph runtime, with purge/embargo protection.
Actual behavior: compute_split_metadata creates split endpoints from ratios, warmup bars, minimum OOS bars, and a required_embargo_gap = max(purge_bars, max_feature_lag + 1). OOS starts at validation_end + required_embargo_gap.  ￼

Temporal validity: The embargo logic is directionally sensible.
Discrepancy / hidden assumption: The embargo gap protects the start of OOS, but it does not fix the training-label boundary leakage discussed above. Also, with current config values (max_lag_bars=16, purge_bars=5, max_cycles=200, minimum_oos_bars=200), a 1000-row dataset would not produce the split counts you supplied; the current code would produce an OOS window much closer to ~182 bars than 199. That suggests either artifact/source drift or an unreviewed code-path mismatch.  ￼  ￼

Model training

Intended behavior: Train only on train split.
Actual behavior: train_lightgbm_baseline slices feature_df[train_start:train_end] and fits on target_up.  ￼

Temporal validity: Not fully valid.
Discrepancy / hidden assumption: Because target_up was precomputed globally, the final training example’s label reaches into the next split. The model therefore does not strictly train only on train information.  ￼  ￼

Validation calibration

Intended behavior: Use validation only for threshold selection.
Actual behavior: prepare_experiment predicts probabilities on the validation slice, then calibrate_thresholds runs Optuna over long_threshold and short_threshold, maximizing validation Sharpe via simulate_policy. OOS data is not passed into calibration.  ￼  ￼

Temporal validity: Mostly yes on data partitioning.
Discrepancy / hidden assumption: The simulation engine used for calibration omits max_drawdown_pause, so the calibrated policy is not the same one run in the live OOS graph. That is a parity defect, not classic leakage, but it weakens the phase-gate evidence materially.  ￼  ￼

Walk-forward folds

Intended behavior: Purged rolling/expanding benchmark before OOS.
Actual behavior: build_walk_forward_folds creates train/validation/test windows before oos_start, inserting purge_bars between validation and test. run_walk_forward_backtest retrains a model on each fold’s train window, calibrates thresholds on fold validation, and simulates on fold test.  ￼

Temporal validity: Partially valid.
Discrepancy / hidden assumption: The purge between validation and test is fine, but the train-label leakage still exists inside each fold, because fold training also uses globally precomputed target_up. So the benchmark is not temporally clean enough to serve as a phase-gate artifact.  ￼  ￼

OOS graph execution

Intended behavior: One-bar-delayed forward simulation over held-out OOS bars only.
Actual behavior: prepare_experiment initializes cursor = oos_start; data_node enforces the minimum OOS cursor; predict_node computes probability and records an execution timestamp at cursor + delay; decision_node queues pending signals until their apply cursor; evaluate_node applies the resulting position to the next bar return. The graph topology is data -> runtime -> predict -> risk -> decision -> evaluate -> optimize.  ￼  ￼  ￼

Temporal validity: The delayed-execution semantics are coherent in the current code, and the tests do meaningfully check runtime-vs-batch parity for this path.
Discrepancy / hidden assumption: Risk is evaluated at signal time, not re-evaluated at execution time. That is defensible for a Week 1 prototype, but it should be explicitly documented because stop-loss / take-profit / volatility blocking are effectively based on the signal bar’s state, not the execution bar’s state. Also, max_drawdown_pause still differs between runtime and simulator.  ￼  ￼  ￼

Metrics and artifacts

Intended behavior: Persist consistent research artifacts sufficient for regression tracking.
Actual behavior: persist_run_artifacts writes report markdown/JSON, dataset and split metadata, strategy params, optimization events, feature importances, trades, completed trades, equity/returns, benchmark metrics, run metrics, readiness, event logs, and CSV exports. It also asserts consistency between report headline KPIs, run metrics, and trade row counts, and rejects legacy trade_count/win_rate keys.  ￼

Temporal validity: Writer logic is strong.
Discrepancy / hidden assumption: The supplied run evidence does not appear consistent with this writer, which means either the artifact bundle was produced by older code, artifacts were modified, or the reviewed source and the assessed run are not actually the same implementation snapshot. That blocks phase-gate confidence.  ￼

7. Hidden or Non-Obvious Issues

The most subtle issue is the global precomputation of labels. Because target_up is added before split slicing, this looks clean at first glance, but it leaks one-bar future information across every train boundary. This is exactly the sort of bug that can survive casual review because the split code itself appears careful.  ￼  ￼

max_drawdown_pause is declared in config and enforced in runtime, but not actually enforced in the simulator used for calibration and walk-forward. That makes it a declared-but-not-consistently-used risk field.  ￼  ￼  ￼

The report diagnostics include an annualized volatility calculation that assumes 1-minute bars, because it uses sqrt(365*24*60) without consulting timeframe. Sharpe annualization is timeframe-aware elsewhere, so the repo currently mixes correct and incorrect annualization conventions across artifacts.  ￼  ￼

The 15m MTF logic is presently okay, but it is fragile under future timeframe changes. The code comments acknowledge that interval defaults are base-timeframe-dependent; if someone changes the base timeframe or adds different MTF intervals, the strictness of the “completed bucket only” semantics should be re-audited.  ￼

evaluate_readiness checks engineering consistency and evidence sufficiency, but it does not treat triple-negative performance itself as a blocker. That is okay if the repo’s goal is engineering readiness rather than strategy readiness, but it means the phase-gate narrative can look overly optimistic unless paired with strict artifact review.  ￼
