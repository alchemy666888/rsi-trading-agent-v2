# Refactor trading agent for timeframe conversion - from 1m to 15m
You are a senior quant/dev engineer doing a careful repo refactor.

Read the repository carefully and do not answer until you have traced all timeframe-sensitive logic end-to-end.

Objective:
Convert this trading research agent so it operates on the **15-minute timeframe by default** instead of the current **1-minute timeframe**, while preserving the existing architecture and keeping timeframe configurable from config.

Your job is not to rewrite the project from scratch. Your job is to make **minimal, correct, production-minded research refactors** that align the entire pipeline to 15m bars.

Core requirement:
The system must be internally consistent after the change:
- market data acquisition
- snapshot filenames / dataset defaults
- feature engineering
- label generation / target definition
- lag logic
- rolling-window logic
- walk-forward splits
- simulation / held-out evaluation
- risk assumptions tied to bar duration
- README / docs / tests

Do not make superficial config-only edits. Audit the whole repo for places where 1-minute assumptions are implicit.

## How to work

1. First, inspect the repo structure and identify all files that are timeframe-sensitive.
2. Then classify each occurrence into one of these buckets:
   - configurable and already safe
   - hardcoded to 1m
   - indirectly calibrated for 1m
   - ambiguous and needs conservative adjustment
3. Then implement the smallest set of changes that makes the repo coherent for 15m trading by default.
4. Keep all changes reversible and config-driven where practical.

## What to change

### 1) Configuration
- Change the default base trading timeframe from `1m` to `15m`.
- Rename default snapshot/data/artifact paths so they no longer imply 1m bars.
- Add or expose config fields wherever timeframe-sensitive behavior is currently buried in code.

### 2) Data ingestion
- Ensure exchange/snapshot loading uses 15m as the default base interval.
- Update helper/fetch scripts so they download 15m candles by default.
- Ensure filenames, cache keys, and output artifacts reflect 15m.

### 3) Feature engineering
Audit all features for hidden 1m assumptions, especially:
- lag counts
- rolling lookbacks
- RSI/EMA/SMA/volatility windows
- resampling logic
- multi-timeframe joins
- any “recent N bars” feature whose real-world duration changes materially when going from 1m to 15m

Rules:
- Preserve intent, not literal bar counts.
- Where a feature was implicitly encoding a time horizon, convert that horizon sensibly to 15m bars.
- Where exact intent is unclear, make the most conservative change and document it.

### 4) Labels / targets
- Ensure the target remains logically correct for the new base timeframe.
- The model should now predict behavior aligned to the **next 15m bar** or equivalent existing target horizon, unless the repo clearly defines a different forecast horizon that should remain fixed in clock time.
- Review whether any thresholding or decision logic assumes 1-minute bar noise or 1-minute execution cadence.

### 5) Training / validation / walk-forward
Audit:
- train/validation/test bar counts
- warmup bars
- optimization horizon
- walk-forward chunk sizes
- evaluation windows
- holdout simulation period definitions

Important:
A raw bar-count carryover from 1m to 15m can accidentally expand or shrink the effective training horizon by 15x. Adjust windows so they remain reasonable in real elapsed time.

### 6) Risk / execution assumptions
Review for any 1m-calibrated assumptions, including:
- stop-loss / take-profit magnitudes
- cooldown periods
- minimum holding periods
- signal delay / action delay
- turnover constraints
- slippage approximations
- bar-close execution assumptions

Do not invent a live execution engine. Stay within the repo’s current research scope.

### 7) Multi-timeframe logic
If the repo joins higher-timeframe features onto the base timeframe:
- verify that these joins still make sense when the base timeframe becomes 15m
- remove redundancy where a formerly “higher timeframe” feature collapses into the new base timeframe
- preserve genuinely useful higher-timeframe context where appropriate

### 8) Documentation and tests
- Update README and inline docs to describe the project as a **15-minute BTC/USDT research trading prototype**
- Update or add tests for timeframe-sensitive behavior
- Ensure examples, commands, and filenames are no longer misleadingly 1m-specific

## Constraints
- Do **not** add live trading, broker APIs, or exchange order execution
- Do **not** redesign the architecture
- Do **not** swap out LightGBM / Optuna / LangGraph unless required to fix a breakage
- Do **not** introduce speculative refactors unrelated to timeframe migration
- Prefer minimal, surgical diffs
- Preserve backward compatibility where easy
- Keep snapshot mode and exchange mode working

## Engineering standard
For every change, prefer this order of operations:
1. make behavior correct
2. make behavior configurable
3. keep diffs small
4. preserve readability

When something is ambiguous:
- inspect neighboring code and infer the original intent
- choose the least risky interpretation
- note the assumption explicitly in your final report
