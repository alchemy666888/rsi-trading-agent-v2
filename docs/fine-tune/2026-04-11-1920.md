You are enhancing the rsi-trading-agent-v2.

Read the repository carefully and do not start to work until you have traced all timeframe-sensitive logic end-to-end.

Context:
- This is a research-oriented BTC/USDT 1-minute trading prototype.
- The runtime graph is roughly: data -> predict -> risk -> decision -> evaluate -> optimize.
- The code already writes artifact bundles per run under `artifacts/<run_id>/`.
- Existing persisted outputs include files like `report.md`, `config.yaml`, `dataset_metadata.json`, `split_metadata.json`, `strategy_params.json`, `optimization_events.json`, `feature_importances.json`, `trades.json`, `trade_history_buffer.json`, `equity_curve.json`, `returns.json`, `benchmark_metrics.json`, `run_metrics.json`, and `shap_rule.txt`.
- Current logging in `run_mvp.py` is minimal and mostly reports completion plus a few summary metrics.

Your task:
Design and implement a much stronger logging and reporting system for this trading agent, while preserving the repo’s current research-prototype character. Do not add live trading or external SaaS dependencies. Keep everything local, file-based, and reproducible.

Goals:
1. Improve observability during runtime.
2. Improve post-run diagnostics and experiment comparison.
3. Make artifacts easier to inspect, diff, and audit.
4. Keep the implementation clean, testable, and lightweight.

Requirements:

A. Structured logging
- Replace or extend current logging with structured logs.
- Use Python logging only, or a very lightweight dependency if absolutely necessary.
- Emit machine-readable JSONL logs to the run artifact directory.
- Also keep a human-readable console logger.
- Every log event should include, where available:
  - timestamp_utc
  - run_id
  - stage or node name
  - event_type
  - symbol
  - timeframe
  - cursor / bar index
  - market timestamp
  - position
  - target_position
  - equity
  - prediction probability / regime
  - risk status
  - thresholds / strategy params
  - message
- Create helper utilities so nodes can log consistently without repetitive boilerplate.

B. Stage-level event logging
Add explicit events for:
- run_started
- config_loaded
- dataset_loaded
- feature_engineering_completed
- model_training_started / completed
- threshold_calibration_started / completed
- walk_forward_started / completed
- oos_simulation_started / completed
- prediction_generated
- risk_evaluated
- decision_made
- trade_opened
- trade_closed
- stop_loss_triggered
- take_profit_triggered
- drawdown_pause_activated / cleared
- optimization_event_recorded
- artifact_persistence_started / completed
- run_completed
- run_failed

C. Trade and decision audit trail
- Create a high-quality execution audit file, e.g. `decision_log.jsonl`.
- For every held-out step, record:
  - bar timestamp
  - close price
  - key features or a compact selected subset
  - model output
  - threshold values
  - pre-risk proposed action
  - risk constraints applied
  - final action
  - resulting position
  - realized / unrealized pnl if available
  - reason codes
- Make it easy to reconstruct why the agent did or did not trade.

D. Enhanced reporting
Upgrade `report.md` into a richer run report, and optionally add `report.json`.
Include:
- run metadata
  - run_id
  - UTC generation time
  - git commit hash if available
  - config hash
  - dataset source and date span
  - total bars / train / validation / OOS counts
- headline KPIs
  - sharpe
  - max drawdown
  - total return
  - win rate
  - trade count
  - avg trade return
  - profit factor if derivable
  - turnover
  - exposure stats
- calibration summary
  - tuned thresholds
  - objective used
  - best validation score
  - optimization trials summary if available
- benchmark comparison
  - held-out metrics vs expanding and rolling walk-forward metrics
  - simple overfit warning heuristics
- risk diagnostics
  - count of blocked trades
  - count of stop loss / take profit events
  - pause durations
  - max consecutive losses
- trade diagnostics
  - best trade
  - worst trade
  - longest hold
  - average hold
  - long vs short breakdown
- data quality / readiness section
  - missing data notes
  - feature availability notes
  - any skipped bars or assumptions
- failure section
  - any exceptions, degraded paths, or fallback behavior encountered

  - drawdown episode extraction
  - event counting by type
  - benchmark comparison
- Keep concerns separated: logging, metrics derivation, artifact persistence, markdown rendering.

G. Error handling
- Make failure paths first-class.
- If a run crashes, still persist a partial artifact bundle with:
  - error summary

  - last known state snapshot
  - partial logs
