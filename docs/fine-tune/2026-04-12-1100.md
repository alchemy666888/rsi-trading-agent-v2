1. Executive Verdict

Do Not Advance

The repository is materially aligned with the intended Week 1 research milestone: it implements heavy BTC feature engineering, a LightGBM directional baseline, validation-only threshold tuning, a held-out LangGraph simulation, a purged walk-forward benchmark, and artifact persistence. The code also stays within the plan’s stated scope and does not drift into RL, fine-tuning, RAG, or live/paper trading claims.  ￼

That said, after tracing the timeframe-sensitive path end to end, I do not think it is phase-gate ready. The main blocker is a simulation-timing semantic defect: the code documents one-bar-delayed / next-bar execution, but both the batch simulator and the LangGraph runtime record execution on a future bar while using the current bar close as the fill price and then immediately earning the next bar return off the updated position. That is not a superficial naming issue; it weakens causal clarity of the held-out and calibration results.  ￼  ￼

I do not see obvious catastrophic lookahead leakage in the core split/feature/target construction. The higher-timeframe features are resampled, then shifted one bucket before backward as-of joining; the target is close.shift(-1) > close; validation is used for threshold calibration; and OOS starts after an embargo gap that exceeds both purge and feature lag. Those are good research-baseline choices.  ￼  ￼  ￼

The current artifact bundle is broad, but artifact semantics are not stable enough for regression tracking. In the checked run, trade_count is consistent between report.md, run_metrics.json, and trades.json, but win_rate is inconsistent because the report headline uses completed-trade statistics while run_metrics.json uses positive-bar-return frequency. That is an audit-grade problem.  ￼  ￼  ￼

Performance is also decisively bad on validation, walk-forward, and held-out metrics. Poor performance alone would not block advancement, but poor performance plus execution-semantics ambiguity plus unstable metric semantics plus snapshot/reproducibility gaps is enough to fail the next-phase gate. The fine-tuning gate in the plan is plainly not satisfied.  ￼  ￼  ￼

2. What Stage the Repository Is Actually In

This codebase is a research baseline with partial reproducibility scaffolding, not merely prototype wiring, but also not yet a reproducible research baseline in the stricter phase-gate sense.

Why that is the right classification:
	•	It has real implementation depth: feature engineering, model training, validation calibration, walk-forward benchmarking, LangGraph held-out execution, artifact persistence, and tests.  ￼
	•	It is still research-only by design and does not claim deployment readiness.  ￼
	•	But the current run is sourced from exchange, not a stable snapshot, and the plan explicitly says future learning work should wait for stable snapshot-based runs and reproducible held-out/walk-forward metrics.  ￼  ￼
	•	The walk-forward benchmark in this run is too thin to function as robust regression evidence.  ￼
	•	Artifact metric semantics are not yet clean enough for audit-grade comparisons across runs.  ￼  ￼  ￼

So the actual maturity level is: usable Week 1 research baseline, not phase-ready for strategy-learning expansion, fine-tuning, paper trading, or production work.

4. End-to-End Timeframe-Sensitive Logic Audit

Data load

Intended behavior: load chronologically ordered BTC/USDT candles, optionally from snapshot, then feature-engineer before splitting.

Actual behavior in code: load_historical_data loads either snapshot or exchange data, sorts by timestamp, injects base timeframe into feature config, builds features, and creates dataset metadata.  ￼

Temporal validity: good.

Discrepancy / hidden assumption: the current run uses dataset.source_mode: exchange, not a frozen snapshot, which is weaker than the plan’s reproducibility gate.  ￼  ￼

⸻

Feature build

Intended behavior: use only information available at or before each prediction bar.

Actual behavior in code: base-timeframe features are standard backward-looking TA and rolling statistics. Higher-timeframe features are grouped with group_by_dynamic(..., closed="left", label="left"), then all MTF feature columns are shifted by one bucket before backward as-of join. That is the correct anti-leakage pattern.  ￼

Temporal validity: strong.

Discrepancy / hidden assumption: after feature construction, all float NaNs/nulls are filled with zero. That is not lookahead leakage, but it can create artificial structure near warmup boundaries and unavailable MTF buckets.  ￼

⸻

Target label

Intended behavior: one-step directional label.

Actual behavior in code: target_up = (close.shift(-1) > close).cast(pl.Int8).fill_null(0).  ￼

Temporal validity: yes.

Discrepancy / hidden assumption: the label is causally valid, but the simulator must match its timing semantics. Right now it does not do so cleanly.

⸻

Split construction

Intended behavior: train for fitting, validation for calibration, embargo before OOS, and OOS held out from fitting/tuning.

Actual behavior in code: _required_embargo_gap is max(purge_bars, max_feature_lag + 1), explicitly accounting for the one-bar label horizon. compute_split_metadata sets oos_start = validation_end + required_embargo_gap and current run metadata shows validation_end=800, required_embargo_gap=17, oos_start=817.  ￼  ￼

Temporal validity: good.

Discrepancy / hidden assumption: oos_end is capped by runtime.max_cycles, so OOS coverage is a runtime-chosen slice rather than full remaining history. That is okay for Week 1, but it affects comparability across runs.  ￼  ￼

⸻

Model training

Intended behavior: fit only on train slice.

Actual behavior in code: train_lightgbm_baseline slices [train_start, train_end) only, builds feature matrix, and fits LightGBM.  ￼

Temporal validity: good.

Discrepancy / hidden assumption: with ~1024 features and only 600 train bars in this run, model variance risk is high.  ￼

⸻

Validation calibration

Intended behavior: use validation only for threshold tuning, not OOS.

Actual behavior in code: prepare_experiment predicts on validation, then calibrate_thresholds runs Optuna against validation-only probabilities/close/regime/timestamps and returns best thresholds plus metadata saying it affects future OOS only.  ￼  ￼  ￼

Temporal validity: OOS contamination is not evident.

Discrepancy / hidden assumption: calibration is consistent with the runtime simulator, but both are consistent to the same flawed execution convention.

⸻

Walk-forward folds

Intended behavior: pre-OOS benchmark with purge gaps between validation and test windows.

Actual behavior in code: build_walk_forward_folds creates train/validation/test windows with test_start = validation_end + purge_bars and stops before oos_start. run_walk_forward_backtest retrains and recalibrates inside each fold.  ￼

Temporal validity: yes.

Discrepancy / hidden assumption: current artifact evidence shows only one expanding fold and one rolling fold. That is benchmark presence, not benchmark sufficiency.  ￼

⸻

OOS graph execution

Intended behavior: forward-only held-out LangGraph simulation with one-bar signal delay.

Actual behavior in code: data_node enforces OOS cursor after validation plus embargo. predict_node computes a future execution timestamp. decision_node schedules pending signals at cursor + delay_bars and, once due, applies them. evaluate_node then uses target_position for the bar return from close_now to close_next, writes trade events with future execution timestamps, but stores price = close_now and sets entry_price = close_now.  ￼

Temporal validity: not clean enough.

Specific mismatch:
	•	batch simulator: scheduled action due at idx, but trade event gets execution_timestamp = timestamp[idx + 1] and price = current_close; then return from close[idx] to close[idx+1] is earned using target_position.  ￼
	•	runtime: applied signal due at cursor, but execution_timestamp is taken from cursor + 1, while trade price and entry price are close_now; then return from close_now to close_next is earned using the updated position.  ￼

That is the highest-severity implementation defect in the repo.

⸻

Metrics and artifacts

Intended behavior: persist complete, internally consistent artifacts for research reproduction and comparison.

Actual behavior in code: run artifacts include report, metrics JSONs, split metadata, thresholds, optimization events, trades, completed trades, returns, equity, and logs. persist_run_artifacts even asserts that headline report trade count must match run_metrics.trade_count, and that trade rows count must also match that same number.  ￼

Temporal validity: persistence is broad, but semantics are inconsistent:
	•	compute_run_metrics.win_rate is based on positive bar returns.  ￼
	•	report headline win_rate is taken from trade_summary.get("win_rate", run_metrics.get("win_rate")), so it prefers completed-trade win rate.  ￼
	•	trade_count is forced to match the number of trades rows, while trade summary statistics come from completed_trades.  ￼

So artifact persistence is broad, but not yet semantically stable enough for regression tracking.

6. Evidence From Current Run Artifacts

The current checked artifact bundle shows strongly negative performance on all three levels that matter:
	•	validation best score: -9.8923 in optimization_events.json and report.md  ￼  ￼
	•	walk-forward overall mean Sharpe: -10.8467 in benchmark_metrics.json  ￼
	•	held-out Sharpe: -46.0520, total return -5.49%, max drawdown 5.49% in run_metrics.json and report.md  ￼  ￼

That implies the baseline is currently failing as a strategy on this sample. By itself, that is still compatible with an honest research outcome. Poor research performance is not the same thing as engineering invalidity.

The more important question is whether the implementation is behaving consistently even if badly. My answer is: partly yes, partly no. It is consistent in the sense that validation calibration, walk-forward benchmark, and held-out runtime all use the same model class and roughly the same simulator logic, so the negative results are directionally coherent. But they are all coherent around a timing model that is not semantically clean, which weakens the interpretability of that consistency.  ￼  ￼

On artifact consistency:
	•	trade_count = 81 matches between report.md, run_metrics.json, and the trade rows written to trades.json; the code explicitly enforces that consistency at persistence time.  ￼  ￼  ￼
	•	But win_rate does not match: report.md shows 47.50%, while run_metrics.json shows 25.82%. That happens because the report prefers completed-trade win rate, while run_metrics.json stores bar-level positive-return frequency.  ￼  ￼  ￼
	•	completed_trades.json contains completed round trips, roughly half the number of trades.json transition events, which is expected — but the schema does not name these differently enough.  ￼  ￼

Also, contrary to the prompt’s supplied summary, the checked artifact evidence does show a readiness/overfit-style warning in the report: “Held-out and walk-forward Sharpe are both negative; strategy is not yet investment-ready,” plus the triple-negative readiness warning.  ￼

7. Hidden or Non-Obvious Issues
	1.	signal_delay_bars=1 is implemented, but only partially in the intended sense. The code does delay when a signal becomes active, but it does not align execution timestamp, fill price, and return attribution onto the same executable bar.  ￼  ￼
	2.	The MTF leakage prevention is actually one of the stronger parts of the repo. The shift-before-join pattern is correct, but fragile: a future contributor could easily remove the one-bucket shift or alter the as-of direction and silently reintroduce leakage.  ￼
	3.	The report says “assumed next-bar execution,” but the implementation does not truly do that. That wording in artifacts.py is therefore misleading.  ￼  ￼  ￼
	4.	Funding logic is timeframe-aware, but still tied to the flawed position clock. funding_interval_bars = 8h / timeframe_minutes is reasonable, but simulator/runtime do not debit the same position state at the same conceptual point in the execution timeline.  ￼  ￼
	5.	max_cycles means OOS length is runtime-capped, not necessarily all available post-embargo history. That is acceptable for Week 1, but risky for run-to-run comparability unless the cap is held constant and explicitly tracked.  ￼  ￼
	6.	The plan says BTC 1m, but actual system assumptions are 15m. Most of the current logic is written around 15m and would need re-audit before timeframe changes.  ￼  ￼
	7.	polygon_api_key is dead configuration in the reviewed path. It is declared but unused by the reviewed data-loading code.  ￼  ￼

	•	split metadata keeps OOS after validation plus embargo  ￼
	•	delayed execution changes timestamps and returns  ￼
	•	runtime and batch simulator agree on returns/equity/trade timestamps under the current semantics  ￼

	•	MTF future spikes do not leak backward  ￼
	•	trade count mismatch in persisted artifacts is explicitly asserted  ￼
	•	readiness triple-negative warning is tested  ￼

What is weakly covered

The tests are weakest exactly where the audit risk is highest:
	•	no test asserts that fill price must come from the actual execution bar
	•	no test asserts clean alignment of signal_timestamp, execution_timestamp, price, entry_price, and realized PnL
	•	no test covers stop-loss/take-profit timing under delayed execution
	•	no test checks funding timing against the correct post-execution position state
	•	no test enforces consistent win_rate semantics across report and JSON
	•	no test enforces distinct meanings for transition-event count vs completed-trade count
	•	no test validates snapshot reproducibility
	•	no test enforces minimum walk-forward sufficiency before calling a benchmark “good enough”

The 10 highest-value additional tests to add next
	1.	Exact delayed-fill-price test
Assert that with signal_delay_bars=1, the stored fill price equals the configured execution bar’s chosen price convention.
	2.	Execution-timestamp/fill-price parity test
Fail if a trade uses a future execution timestamp with a current-bar fill price.
	3.	Runtime-vs-batch full trade-log parity test
Compare action, from_position, to_position, signal_timestamp, execution_timestamp, price, entry_price, and completed-trade outputs.
	4.	Stop-loss timing test
Confirm exactly which bar closes the position and what price/timestamp is recorded.
	5.	Take-profit timing test
Same as above.
	6.	Funding timing test
Verify that funding is charged on the intended position state at the intended bar cadence for 15m.
	7.	Metric-schema reconciliation test
Assert that report and JSON either match exactly or intentionally use separately named metrics.
	8.	Completed-trade-count vs transition-count test
Explicitly store and validate both.
	9.	Snapshot reproducibility test
Same snapshot + same config + same commit should reproduce thresholds and metrics.
	10.	Benchmark sufficiency test
Require minimum fold count or emit an insufficiency flag.

9. Phase-Gate Recommendation

The project should remain in the current phase until material research-quality issues are resolved.

The next phase should be defined narrowly as: a reproducible research baseline with audit-clean temporal semantics, stable artifact schemas, and benchmark regression readiness across multiple historical windows. It should not mean RL, adaptive learning loops, fine-tuning, live trading, or paper trading.  ￼

Must-fix before advancing
	•	Correct execution timestamp / fill-price / return-attribution semantics in both simulator and LangGraph runtime.  ￼  ￼
	•	Normalize KPI semantics so bar-level and completed-trade-level metrics are distinct and consistently named.  ￼  ￼
	•	Move benchmarked runs to snapshot-based reproducible inputs.  ￼  ￼
	•	Add benchmark sufficiency checks or increase historical windows/folds.  ￼

12. Final Bottom Line

This repository is a real Week 1 research baseline, not just mock wiring, and its anti-leakage split/feature design is better than average for an early prototype. But it is not good enough to move forward yet. It should remain in the current phase until execution timing is made causally consistent, KPI semantics are normalized across artifacts, snapshot-based reproducibility is enforced, and walk-forward evidence becomes benchmark-sufficient. Until those are fixed, the correct decision is Do Not Advance.
